- **Bug #7** 🟡 `recordings/start` body type-coercion crash — `body.account_id?.trim()`

- **Bug #25** 🟡 Screenshot data URL was unreadable by Claude Code agent's

- **Bug #27** 🟢 agent-skill description now matches Bug #25 reality

### Phase E — security (cycle 5-7)

- **Bug #1** 🔴 **CRITICAL** Selector injection (3 sites). `replace(/'/g, "\\'")`
  could be bypassed by `\\'` (backslash + quote). Any malicious selector
  returned by `agent_repair` could break out of the JS string in

- **Bug #5** 🔴 **CRITICAL** SQL filter injection in pipeline-status `.or()`.

- **Bug #9** 🟢 Agent prompt injection — defense-in-depth notation only

## The 5 critical finds — one paragraph each

### Bug #20 — async pipeline never ran in prod

This one would have ruined the entire feature on day 1 of merge. The
recording UI calls `/api/recordings/stop`, which triggers `runPipelineAsync()`
to fire-and-forget POST to `/analyze` → `/build-automation` → `/self-test`.
Each of those routes is gated by the same `withAuth()` middleware as
everything else. The internal `fetch()` calls had no `Cookie:` header,
so middleware 401'd them silently. The modal's progress poller would
have waited forever for status to advance past "analyzing…". On a
sweep cycle this surfaced as "every recording stuck at 25%". Fix:
capture the cookie before sending the response, forward it on every
internal fetch.

### Bug #2 — modal could never show Recording state

Pure React-trap. `useDummySelection()` returned a fresh object literal
every render (`{ accountId, proxyId, ... }`). The recording modal had
a useEffect that reset state when the dummy changed, with `dummy`
in the deps array. Object identity changes every render, so the
effect ran every render, immediately resetting `isRecording` back
to false. Click Start Recording → state goes true for one tick →
effect runs → state goes false. Modal flickers and never enters
recording state. Fix: drop `dummy` from deps with a comment
explaining why. The kind of bug only a beating-it-up audit catches.

### Bug #5 — SQL filter injection in pipeline-status

The `/api/recordings/[id]/pipeline-status` route did
`.or('recording_id.eq.${id},automation_id.eq.${id}')` against
PostgREST. The `${id}` was interpolated raw — a malicious id of
`abc),other_filter:dangerous` would break out of the filter clause.
Even though `id` was UUID-shaped in the path, no validation enforced
that. Fix: UUID-validate at the route boundary + use parametrized
`.in()` arrays instead of raw `.or()` strings.

### Bug #1 — Selector injection (3 sites)

The self-test route runs `Runtime.evaluate` against the live Chrome
instance with selectors that ultimately come from agent-repair output.
The escape function was `replace(/'/g, "\\'")` — which can be bypassed
by `\\'` (backslash escaping the backslash, then a literal quote).
A poisoned agent response containing `\\'; DROP TABLE accounts; //`
(or worse, exfil JS) would execute in the target page context.
Switched all 3 sites to `JSON.stringify(selector)` interpolation.

### Bug #26 — Proxy credential leak

`/api/automations/dummy-selection` GET endpoint joined `proxy_groups`
to surface the proxy assigned to each dummy account, and the SELECT
list included `username, password`. Any authenticated dashboard user
hitting `GET /api/automations/dummy-selection` got proxy creds in
the response body. This shipped behind the PIN gate so it's not
"public on the internet" bad — but every operator sharing the
dashboard would have proxy creds in their devtools network tab.
Fix: remove the credential columns from the SELECT list.

## Lessons (file these in feedback memory)

- **Polling matters.** The audit's full-sweep harness loops 27 combos
  × 4 runs = 108 lifecycles per cycle. That kind of repetition is
  what surfaces the "looks fine in dev tools but broken in prod"
  bugs like #20. One-off Playwright tests would not have caught
  this.
- **State reset useEffects with object dependencies are a footgun.**
  Bug #2 is the canonical case. If a hook returns a new object each
  render, that object cannot go in deps unless you memo it. Lint
  rule could catch this.
- **Internal fetch() between Vercel routes does NOT inherit auth.**
  Cookie has to be forwarded explicitly. This is the second time
  this has bit us. Adding it to the lessons memory tree.
- **Object literal ↔ deps array** is a widespread pattern issue —
  worth a lint rule pass across the codebase next time we have
  a calm session.
